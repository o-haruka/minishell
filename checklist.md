## 1. 基本機能 (Basics)
- [ ] **プロンプト表示**: 新しいコマンド入力を待つ際、プロンプトが表示される。
- [ ] **履歴機能**: 実行したコマンドの履歴（History）が機能する。
- [ ] **実行ファイルの探索**: 
    - [ ] `PATH` 変数に基づく検索・実行。
    - [ ] 相対パス（`./script.sh` 等）による実行。
    - [ ] 絶対パス（`/bin/ls` 等）による実行。

## 2. シグナル処理 (Signals)
- [ ] **グローバル変数の制約**:
    - [ ] グローバル変数は最大1つのみ。
    - [ ] 受信したシグナル番号の保持のみに使用。
    - [ ] シグナルハンドラがメインのデータ構造（構造体など）に直接アクセスしていない。
- [ ] **インタラクティブモードの挙動**:
    - [ ] `Ctrl-C`: 新しい行に新しいプロンプトを表示する。
    - [ ] `Ctrl-D`: シェルを終了する。
    - [ ] `Ctrl-\`: 何もしない（bashの挙動に準拠）。
> Ctrl+C (SIGINT) や Ctrl+\ (SIGQUIT) を受け取った際に、プロンプトがどう振る舞うべきか（新しい行を表示する、入力をクリアするなど）を、本物の bash の挙動と比較しながらテストすることをおすすめします。

> ③ シグナルとグローバル変数
> volatile sig_atomic_t 型を使用して、シグナル番号だけを保持する設計にします。
> 
> ハンドラ内で printf や malloc などの非安全な関数を使わず、メインループに「シグナルが来た」ことを伝えるだけに留めるのが安全です。

## 3. 解析・パース (Parsing)
- [ ] **クォート処理**:
    - [ ] **シングルクォート `' '`**: 囲まれた中のメタ文字を解釈しない。
    - [ ] **ダブルクォート `" "`**: 囲まれた中のメタ文字を解釈しないが、`$` (変数展開) のみ解釈する。
- [ ] **非対応文字の無視**: 
    - [ ] 要求されていない `\` (バックスラッシュ) や `;` (セミコロン) を解釈しない。
        
         （補足：bashでは \ で改行をエスケープしたりできますが、この課題では「何もしなくて良い」という意味です。閉じられていないクォートもエラー処理や特別な挙動をさせる必要はありません。）
    - [ ] 閉じられていないクォートを厳密に処理する必要はない。（" が1つだけ等）に対して、特別な多行入力モード（> プロンプト）などを実装しない。

> クォートの処理を実装する際は、文字列を1文字ずつスキャンしながら「今はどの状態（クォート外、シングルクォート内、ダブルクォート内）にいるか」を管理する状態遷移図をイメージするとバグが減ります。
> 
> 特に、" $USER " のように「ダブルクォートの中に変数展開がある」ケースと、' $USER ' のように「シングルクォートの中なので変数を展開しない」ケースの切り分けがテストでよく狙われるポイントです。

> ① パーサー（字句解析・構文解析）
> 入力された文字列をトークン（単語、パイプ、リダイレクト）に分解し、実行可能な形に構造化します。
> 
> クォートの除去: "ls" も ls も同じコマンドとして扱えるよう、実行前にクォートを取り除く必要があります。
> 
> 展開の順序: 環境変数の展開（Expand）は、クォートの処理と密接に関係します。

## 4. 環境変数と展開 (Expansion)
- [ ] **変数展開**: `$VARIABLE` をその値に展開する。
- [ ] **終了ステータス**: `$?` を直近のパイプラインの終了ステータスに展開する。

> 環境変数の展開 " $VAR "（ダブルクォート内）では展開が必要だが、' $VAR '（シングルクォート内）では展開してはいけない。先にパースして「クォートの状態」を確定させてから展開処理を行うのが一般的。

> 環境変数のコピー main 関数の envp はプログラム開始時のスナップショットです。export や unset を実装するためには、起動時に envp の内容をリスト構造や動的配列にコピーして保持し、それを自前で管理する必要があります。

> ④ 環境変数の管理
> main の第3引数 char **envp は読み取り専用と考え、export/unset ができるように自分でコピーしたリスト（または配列）を管理します。

## 5. リダイレクトとパイプ (Redirections & Pipes)
- [ ] **リダイレクト**:
    - [ ] `<` : 入力リダイレクト。
    - [ ] `>` : 出力リダイレクト（上書き）。
    - [ ] `>>`: 出力リダイレクト（追記）。
    - [ ] `<<`: ヒアドキュメント（履歴には保存しない）。
- [ ] **パイプ `|`**:
    - [ ] 各コマンドの出力を次のコマンドの入力へ接続する。

> パイプとリダイレクトの構造 パイプとリダイレクトを組み合わせる際、dup2() や pipe() のシステムコールを使うタイミングが重要です。特に、複数のパイプがある場合のファイル記述子（FD）の管理は複雑になりやすいため、図解などで整理することをおすすめします。

> ② プロセス制御とパイプ
> fork, execve, pipe, dup2, waitpid を駆使します。
> 
> FD（ファイル記述子）管理: pipe() で作ったFDを dup2() で 0(stdin) や 1(stdout) に繋ぎ変えます。不要になったFDを即座に close しないと、プロセスが終了せず停止（ハング）します。

## 6. ビルトインコマンド (Built-ins)
- [ ] `echo` (オプション `-n` 対応)
- [ ] `cd` (相対パス・絶対パスのみ)
- [ ] `pwd` (オプションなし)
- [ ] `export` (オプションなし)
- [ ] `unset` (オプションなし)
- [ ] `env` (オプション・引数なし)
- [ ] `exit` (オプションなし)

## 7. メモリ・品質管理
- [ ] **メモリリーク**:
    - [ ] 自分が作成したコード内にリークがない（`readline` 本体のリークは不問）。
- [ ] **仕様の遵守**: 要求されていない機能は実装せず、迷った際は `bash` の挙動を正解とする。
